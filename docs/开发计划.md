## 二、开发阶段

### 阶段一：脚手架与基础架构（Day 1 上午）

**目标：** 搭建完整目录结构，注册 Manifest V3，实现插件可在 Chrome 中加载。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 1.1 | 创建完整目录结构 | 上述目录树所有文件占位 |
| 1.2 | 编写 `manifest.json` | 声明权限（storage, activeTab, scripting）、注册 popup/options/background/content |
| 1.3 | 生成占位图标 | 16/48/128 尺寸简单 SVG 转 PNG 图标 |

### 阶段二：网页内容抓取（Day 1 下午）

**目标：** 内容脚本能智能抽取网页正文，弹窗能获取并展示。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 2.1 | 编写 `content/content.js` | 抽取标题 `<title>` / `og:title` → URL `location.href` → 正文（Readability 原理：取 `<article>` / 最高密文容器，过滤 nav/header/footer/aside/.ad 等） |
| 2.2 | 实现智能正文提取 | 按文本密度算法：遍历 body 子元素，选择文本密度最高的容器作为正文 |
| 2.3 | 内容净化过滤 | 删除不可见元素、脚本、样式、广告特征 class/id |
| 2.4 | 内容截断 | 超出 8000 字符时截断 + 标记"内容过长已截断" |
| 2.5 | popup ↔ content 通信 | popup 打开时通过 `chrome.tabs.sendMessage` 请求抓取，content 返回 { title, url, content } |
| 2.6 | 实现 `popup/popup.html` 骨架 | 三个分区：基础信息区、标签区、操作按钮区 |
| 2.7 | 实现 `popup/popup.css` 基础样式 | 极简设计，亮色主题（深色待阶段五），flex 布局适配弹窗尺寸 |
| 2.8 | 弹窗展示抓取结果 | 显示标题、URL（可截断）、抓取状态 |

**测试要点：**
- 普通文章页能否正确提取正文
- SPA 页面能否获取内容
- 无正文页面（如控制台）的降级处理

### 阶段三：AI 标签生成（Day 2 上午）

**目标：** 对接 LLM API，实现 5 维度智能标签生成并展示。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 3.1 | 编写 `lib/api.js` | 封装 fetch 调用 LLM API（支持 OpenAI 兼容格式），含超时控制、错误分类 |
| 3.2 | 编写标签生成 Prompt | 系统 Prompt 包含 5 维度定义、输出格式要求（JSON）、单标签 2-6 字限制、去重要求 |
| 3.3 | 解析 LLM 返回 | 将模型返回的 JSON 解析为 5 维度标签数组，处理格式异常 |
| 3.4 | 智能去重后处理 | 同维度内去重、合并同义词标签 |
| 3.5 | `popup/popup.js` 标签渲染 | 按 5 维度分组展示标签，每个标签 chips 样式，可删除 |
| 3.6 | 添加标签功能 | 输入框 + 回车，手动新增标签到指定维度 |
| 3.7 | 编辑标签功能 | 点击标签可编辑文字（inline edit） |
| 3.8 | 加载动画 & 失败提示 | API 请求中显示 skeleton 加载态，失败显示具体错误描述 + 重试按钮 |

**Prompt 核心设计：**

```
你是一个网页内容分析专家。请分析以下网页内容，按 5 个维度生成标签。

维度说明：
1. 内容领域 — 网页所属的知识/行业领域，如 "科技"、"教育"、"金融"
2. 用途场景 — 适合什么场景使用，如 "学习参考"、"工作文档"、"休闲阅读"
3. 难度等级 — 内容的专业程度，只能选 1 个：入门 / 进阶 / 专业
4. 核心关键词 — 2-4 个代表性关键词，如 "AI"、"机器学习"
5. 内容属性 — 网页的类型，如 "教程"、"新闻"、"分析"、"工具"

要求：
- 每个标签 2-6 个字，简洁标准
- 同维度内自动去重、合并同类项
- 以 JSON 格式输出：{ "contentDomain":[], "usageScenario":[], ... }
- 除了 JSON 不要输出任何其他内容

网页内容：{content}
```

### 阶段四：本地存储与历史管理（Day 2 下午）

**目标：** 标签数据持久化，支持历史记录浏览与管理。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 4.1 | 标签保存流程 | 编辑完成 → 自动保存至 storage.local（URL 为 key） |
| 4.2 | URL 覆盖逻辑 | 相同 URL 不重复创建，直接更新 updatedAt + tags |
| 4.3 | 历史记录列表视图 | 弹窗内切换至历史列表，按 updatedAt 倒序排列 |
| 4.4 | 历史条目操作 | 点击回填至编辑视图、单条删除（带确认）、清空全部（带二次确认） |
| 4.5 | 空状态处理 | 无历史记录时显示"暂无标签记录"引导文案 |
| 4.6 | 自动保存防抖 | 编辑操作 500ms 防抖后触发保存，避免频繁写入 |

### 阶段六：异常处理与性能优化（Day 3 下午）

**目标：** 全覆盖异常场景，打磨体验细节。

| 步骤 | 内容 | 产出 |
|------|------|------|
| 6.1 | 网络异常处理 | fetch timeout（15s）、网络断开、HTTP 非 200 等场景统一错误提示 |
| 6.2 | API 层错误分类 | 401 → "API Key 无效，请检查设置"；429 → "请求过于频繁，请稍后重试"；其他 → "服务异常，请重试" |
| 6.3 | 空内容处理 | content 抓取为空 → 提示"当前页面无可抓取内容"，AI 标签区域隐藏 |
| 6.4 | 抓取节流 | 同一 tab 30s 内不重复抓取，按钮显示倒计时 |
| 6.5 | 内存优化 | 弹窗关闭时清理定时器、事件监听，释放图片/大文本引用 |
| 6.6 | 全局入口校验 | 每次弹窗打开检查 API Key 是否已配置，未配置则引导至设置页 |
| 6.7 | 最终集成测试 | 全功能流走一遍：打开网页 → 抓取 → AI 生成 → 编辑 → 保存 → 历史查看 → 清空 |

## 四、风险与备选方案

| 风险 | 概率 | 影响 | 备选方案 |
|------|------|------|---------|
| 正文提取不准确 | 中 | 中 | ① 引入 mozilla/readability 算法（纯 JS 轻量版）② 提供手动选择区域功能（P2） |
| 不同 LLM API 返回格式不一致 | 中 | 高 | 所有适配器统一输出格式；模型返回异常时重试 1 次 |
| Manifest V3 下 content script 通信限制 | 低 | 中 | 使用 chrome.scripting.executeScript 动态注入替代静态 content_scripts |
| API Key 存储安全 | 中 | 高 | XOR 加密 + 不在页面 DOM 中暴露；隐私政策说明数据仅存本地 |

## 六、开发原则

1. **模块优先** — 每个 JS 文件一个职责，避免 spaghetti code
2. **渐进增强** — P0 功能先跑通，P1/P2 逐步打磨
3. **用户可见交付** — 每完成一个阶段，功能可操作、可验证
4. **零外部依赖** — 纯原生实现，确保开箱即用
5. **错误友好** — 所有异常都有用户可理解的提示，而非 console.error 了事
